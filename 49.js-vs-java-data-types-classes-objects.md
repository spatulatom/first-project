## Intro

(ang. b-code, byte-code), a następnie interpretowany przez wirtualną maszynę Javy (ang. Java Virtual 
Machine, JVM) pozwalającą na uruchomienie aplikacji na różnych maszynach. 
Język realizujący paradygmat programowania zorientowanego obiektowo - w którym programy definiuje 
się za pomocą obiektów – elementów łączących stan (czyli dane, nazywane najczęściej atrybutami) 
i zachowanie (czyli procedury, tu: metody). Obiektowy program komputerowy wyrażony jest jako zbiór 
takich obiektów, komunikujących się pomiędzy sobą w celu wykonywania zadań. 
Programowanie obiektowe ma ułatwić pisanie, konserwację i wielokrotne użycie programów lub ich 
fragmentów.

When comparing programming languages, there are actually **two separate distinctions** worth keeping apart — and they are independent of each other.

### First distinction — what does compilation produce?

Every language gets compiled eventually. The question is *what the compilation step outputs*:

- **Intermediate bytecode** (needs a VM to execute): Java (`.class` files → JVM), C# (IL → .NET CLR), Python (`.pyc` → CPython). The source compiles to a portable, platform-independent format — but one that still needs a runtime to turn it into real machine instructions.
- **Native machine code** (runs on the CPU directly, no VM needed): C, C++, Rust, Go. No runtime dependency; the output binary is immediately executable.

JavaScript is an outlier that doesn't fit either bucket cleanly: there is no intermediate bytecode layer you see, and no native binary you produce. The source itself is handed to an engine, which JIT-compiles it to machine code invisibly at runtime.

### Second distinction — when does compilation happen? (AOT vs JIT)

Even knowing what compilation produces, there's a further question: does that compilation happen *before* execution (at build time), or *during* execution (at runtime)?

| Term    | Full name     | When it happens               | Examples                                          |
| ------- | ------------- | ----------------------------- | ------------------------------------------------- |
| **AOT** | Ahead-Of-Time | Build time, before anything runs | C, C++, Rust, `javac` → bytecode, GraalVM native |
| **JIT** | Just-In-Time  | Runtime, as code executes     | JVM executing bytecode, V8 running JS             |

These two distinctions are **independent**. C and C++ are AOT and compile to native machine code. Java's `javac` step is AOT but compiles to intermediate bytecode — then the JVM JIT-compiles that bytecode to native code at runtime. JavaScript skips the intermediate form entirely and goes straight from source to JIT-compiled native code. The *target* of compilation and the *timing* of compilation are separate questions.

### A note on "build time" — does it always exist?

The AOT/JIT framing talks about "build time" as if it's always present, but that's not always true — and JavaScript is the clearest example:

- **Java** — build time always exists. `javac` is mandatory. You cannot run Java without first running the compiler. Build time is baked into the language, not optional.
- **Vanilla JavaScript** (no bundler, no tooling) — there is no build step at all. You write `.js` files, open a browser or run `node app.js`, and the code executes. The JIT happens entirely inside the engine, invisibly. From the developer's perspective there is only edit time and runtime — no build time.
- **Modern JavaScript** (with Vite, Next.js, webpack, etc.) — `npm run build` exists, but this is *transpilation and bundling* added by tooling, not something the JavaScript language requires. The engine's JIT still happens later, at runtime. These are two separate steps: one imposed by your toolchain, one done by the engine.

So "build time" is not a universal concept. For AOT languages it is unavoidable — it *is* the compile step. For JIT languages it only exists if you choose to add tooling. Vanilla JS has no build step at all, and it still runs fine; the JIT does its job silently.

### For AOT languages, can a developer overlook compilation?

No — and this is the key asymmetry. For AOT languages, the developer *has* to set up the compile step deliberately. With Java you invoke `javac` (or configure Maven/Gradle to do it); with C/C++ you run `gcc`/`clang` or write a Makefile; with Rust you run `cargo build`. There is no path to running the code that bypasses this. The compilation gate is something every developer in those ecosystems must understand and configure from day one.

For languages that compile to native machine code (C, C++, Rust), the build setup is if anything *more* involved than Java — not less. Java bytecode is platform-independent: compile once, run on any machine that has a JVM. Native code is not: you compile for a specific OS and CPU architecture, so the build step must also know its *target*. Moving that binary to a different platform (say, Windows to Linux) requires recompiling. Cross-compilation — building for a target platform different from the one you're building on — adds another layer of toolchain configuration. Rust's `cargo` smooths a lot of this over, but it's still explicit. C/C++ with `Makefile`s or CMake can get quite involved.

So native AOT languages don't simplify the build step compared to Java — they complicate it further. Java trades that complexity for a JVM dependency; you compile once and the JVM handles the platform differences at runtime.

This is the opposite of JavaScript, where a beginner can write code for months without ever knowing compilation happens at all.

### Is JavaScript exceptional in how invisible its compilation is?

Mostly yes, but not completely alone. Other JIT-compiled languages — Python, Ruby — also hide compilation from the developer. You don't invoke a compiler to run a Python script either. The difference is that Python at least leaves a visible trace: `.pyc` files appear in `__pycache__/`, hinting that something was compiled. JavaScript leaves no trace whatsoever — no intermediate files, no artefacts, nothing. The JIT output lives in memory, executes, and disappears.

So JavaScript is not the only language with invisible JIT compilation, but it is one of the cleanest examples of it — there is genuinely nothing for a developer to see or set up, even if they go looking.

From a **developer experience (DX)** standpoint, the second distinction (AOT vs JIT) arguably matters more day-to-day than the first:
- AOT means errors are caught before you run anything — the compiler reads your whole program and fails fast
- JIT means errors only surface when that specific code path actually executes

The first distinction (what bytecode/native code the compiler produces) is often closer to a technical implementation detail — relevant if you care about portability, runtime dependencies, or startup performance, but not something you feel on every save. The second one — whether mistakes surface at build time or only at runtime — is something you feel constantly.

### Are there tell-tale signs a developer can see, without knowing what's under the hood?

For the **second distinction (AOT vs JIT)**: yes, very clearly.

- **AOT** — you have to run a separate compile command *before* you can execute anything. In Java: `javac MyFile.java` must succeed and produce `.class` files; only then can you do `java MyFile`. If `javac` fails, nothing runs at all. The compile step is a gate.
- **JIT** — there is no explicit compile step you trigger. In JavaScript: you just run `node app.js` or open a browser. The code executes immediately; there is nothing to produce first.

The tell-tale sign: *does the language stop you from running if something is wrong before you even try?* AOT languages do. JIT languages don't — they fail only when the bad line actually executes.

For the **first distinction (what compilation produces)**: visible, but indirectly.

- **Bytecode/VM languages** (Java, C#, Python): you see intermediate artefact files appearing — `.class`, `.pyc`, compiled IL. You also need a specific runtime installed on any machine that runs the code (JVM, .NET CLR) — the files don't run on their own.
- **Native languages** (C, C++, Rust): you get a standalone binary. No runtime dependency. You compile once for a specific OS/CPU and that binary just runs. Moving the binary to a machine with no tooling installed still works.
- **JavaScript**: you see no output artefacts at all from your own code. Nothing gets produced; the source files *are* the thing you ship. The runtime (browser or Node) is bundled into the environment rather than something you install separately.

So for Java vs JavaScript specifically: the AOT/JIT split (second distinction) is something every developer bumps into on day one. The bytecode vs JIT-only split (first distinction) surfaces later — when you notice `.class` files appearing, or ask why Java needs a JVM installed but a compiled Rust binary doesn't.

## Key Insights — Compilation, Static Checking & Coverage

### Java is both AOT and JIT — not purely one or the other

A common misconception is that Java is AOT-compiled and JavaScript is JIT-compiled, as if they are opposites. The truth:

- **JavaScript** — JIT only. Source → JS engine → machine code at runtime.
- **Java (default JVM mode)** — AOT _and_ JIT. `javac` compiles source to bytecode first (AOT), then the JVM JIT-compiles bytecode to machine code at runtime.
- **Java (GraalVM Native Image)** — true full AOT, producing a native binary with no JVM and no JIT.

Java feels "more compiled" because it has a mandatory upfront step. But that step produces bytecode, not native machine code — the JIT still happens later.

The practical consequence of the AOT step is unchanged: type errors are caught before execution. Whether the output is bytecode or native code doesn't affect when errors surface for the developer.

### What compilation is actually useful for

Compilation (the upfront static analysis step) is not mimicking execution — it does something execution cannot: it **reads the entire program at once** and checks whether type contracts are consistent, without needing any real data or actual execution.

When code _executes_, it only runs the paths triggered by real inputs — bugs on untouched paths stay hidden.  
When a compiler _analyzes_, it checks **all** paths regardless of whether they ever run.

|          | What it catches                                                 |
| -------- | --------------------------------------------------------------- |
| Compiler | "This code cannot possibly work" (structural/type errors)       |
| Tests    | "This code doesn't do what I intended" (logic/behaviour errors) |

Static typing does not eliminate the need for tests — logic errors, runtime conditions, and integration bugs are invisible to a compiler.

### You can mimic static checking in JavaScript — but it's opt-in

Nothing stops a JS/TS team from running `tsc --noEmit` in CI to get the same upfront type-checking Java forces automatically. The difference is enforcement:

|                  | Static checking                                                    |
| ---------------- | ------------------------------------------------------------------ |
| Java             | Mandatory — `javac` must pass before anything runs                 |
| TypeScript       | Opt-in — `tsc --noEmit` in CI, bypassable with `any` / `ts-ignore` |
| Plain JavaScript | None — no type annotations, nothing to check                       |

`tsc --noEmit` tells the TypeScript compiler to type-check the whole codebase but produce no output files — useful when SWC/esbuild is already handling the JS output faster.

### JIT error coverage vs static analysis coverage

JIT compiles (and therefore can only surface errors in) code that **actually runs**. Static analysis covers the **whole program by default**.

To get equivalent coverage from JIT alone, you'd need 100% code path coverage from tests and real usage — practically impossible and expensive to maintain.

> JIT error coverage = however much of your code actually ran.  
> Static analysis error coverage = the whole program, always.

## Compiled vs Interpreted — The Root Difference

Your instinct is right — this is arguably the **most fundamental** difference, and most other differences do follow from it.

### Java: Compiled

Java code goes through a **compile step** before it ever runs:

1. You write `.java` source code
2. The compiler (`javac`) reads the whole program and checks it for errors
3. It produces `.class` bytecode files
4. The JVM runs the bytecode

Because the compiler reads the entire program upfront, it can **enforce rules before anything executes**. This is why Java requires you to declare types — the compiler needs to verify that you're not passing a `String` where an `int` is expected. Catching that mistake at compile time is the whole point.

### JavaScript: Not Compiled (in the traditional sense)

JavaScript was designed to run directly in a browser — there is no separate compile step you trigger. Code is handed to the engine and executed immediately. This means:

- There's no upfront analysis checking your whole program
- Types can't be enforced before runtime — the engine doesn't know what type a variable will be until it actually runs that line
- Flexibility is necessary, not optional

> **Important caveat:** Modern JS engines (like V8 in Chrome/Node) do use JIT (Just-In-Time) compilation internally for performance — but this is invisible to you. It's not the same as Java's explicit compile step, and it doesn't add type checking.

### How This Cascades Into Every Other Difference

| Root cause                       | Java consequence                                         | JavaScript consequence                           |
| -------------------------------- | -------------------------------------------------------- | ------------------------------------------------ |
| Compile step exists              | Static typing enforced at compile time                   | No static typing — types checked at runtime only |
| Types known at compile time      | Variables must be declared with a type                   | `var`/`let`/`const` — no type needed             |
| Compile-time type safety         | Real classes as a compile-time construct                 | No real classes — prototypes work at runtime     |
| Type system needs completeness   | Wrapper classes to make primitives fit the object system | Auto-boxing at runtime when methods are needed   |
| Strict contract before execution | Strong, safe, verbose                                    | Flexible, loose, concise                         |

### The One-Sentence Version

> Java is designed around a compiler that checks everything before running, so it _requires_ static types and real classes.  
> JavaScript is designed to run immediately without a compile step, so it _has to be_ dynamic, loose, and prototype-based.

TypeScript is the bridge — it adds a compile step (and static types) on top of JavaScript, which is why TypeScript code looks much more like Java.

### AOT vs JIT — the precise terms

The standard terminology for the Java vs JavaScript compilation model is:

- **AOT (Ahead-Of-Time)** — compile _before_ execution. Java's approach.
- **JIT (Just-In-Time)** — compile _during_ execution. JavaScript's approach.

But "before execution" and "during execution" still need more precise labels, because "development" and "production" don't map cleanly — you can run code in development (that's runtime), and you can run a build step in a production CI pipeline (that's build time). Those words are too broad.

The correct terms are **build time** and **runtime**:

| Term           | Meaning                                                                                                     |
| -------------- | ----------------------------------------------------------------------------------------------------------- |
| **Build time** | The phase when source code is transformed before any execution occurs — running `javac`, `tsc`, `swc`, etc. |
| **Runtime**    | When the code actually executes — in a browser, on a server, anywhere                                       |

So the cleanest framing:

> **AOT** = compilation happens at **build time** (before runtime)  
> **JIT** = compilation happens at **runtime** (during execution)

This is also why Java catches type errors at _build time_ and JavaScript only catches them at _runtime_ — the compile step simply happens at different moments relative to execution.

### Edit time — the phase before build time

There is actually a third phase that sits even before build time, and coming from TypeScript/VSCode you experience it constantly: **edit time** (also called **author time**).

When VSCode shows you a red squiggle as you type — before you've run `npm run build` or anything — that's the **TypeScript Language Server** (`tsserver`) running in the background. It continuously type-checks your code incrementally as you edit, giving you immediate feedback.

This is _not_ the TypeScript compiler (`tsc`) producing output files. It is a separate tool designed purely for editor integration.

Java IDEs do the same thing — IntelliJ and Eclipse run the Java compiler in the background as you type, showing errors in real time. It feels like the language itself, but it's actually the IDE's language server tooling.

**The three phases:**

| Phase          | Trigger                 | Tool (TypeScript)            | Tool (Java)                            |
| -------------- | ----------------------- | ---------------------------- | -------------------------------------- |
| **Edit time**  | As you type             | `tsserver` (language server) | IntelliJ / Eclipse background compiler |
| **Build time** | `npm run build` / `tsc` | `tsc` / `swc` / `next build` | `javac`                                |
| **Runtime**    | Code executes           | V8 JIT in browser/Node       | JVM JIT                                |

So your instinct to reach for "development" was pointing at edit time — a real and distinct phase. "Development" was just too broad a word for it, since it also covers running the app locally (which is runtime).

> **Edit time** errors = language server catches them as you type  
> **Build time** errors = compiler catches them when you explicitly build  
> **Runtime** errors = only caught when the code actually executes

## Compilation vs Execution — Not the Same Thing

These two steps are easy to conflate, but they are completely separate:

- **Compilation** — transforming source code into another form (bytecode, machine code, or even another language)
- **Execution** — actually running the result of compilation

They can happen far apart in time (Java: you compile once, distribute the `.class` files, run later) or almost simultaneously (JS in the browser: the engine compiles and runs in milliseconds). But they are always logically distinct steps.

### What compilation produces is not human-readable

This is the key point: **the output of real compilation is not source code**. It is a lower-level representation designed for machines, not people.

- Java's `javac` produces **bytecode** (`.class` files) — a binary format the JVM understands, not you
- The JVM's JIT compiler turns bytecode into **native machine code** — CPU instructions, entirely unreadable to humans
- JavaScript in the browser: V8 (Chrome/Node), SpiderMonkey (Firefox), and JavaScriptCore (Safari) all JIT-compile your JS to **machine code** internally before running it

So even though you never trigger a compile step yourself in JS, compilation still happens — inside the engine, invisibly, every time your script runs.

### The full pipeline for each language

**Java:**

```
.java source → [javac compiler] → .class bytecode → [JVM JIT] → machine code → CPU executes
```

**JavaScript in the browser:**

```
.js source → [V8 JIT compiler, invisible] → machine code → CPU executes
```

The difference is not _whether_ compilation happens — it's _when_ and _who controls it_.

### Are .class files human-readable? (Correcting a common misconception)

No — `.class` files are **binary bytecode**. Open one in a text editor and you get unreadable binary noise. The reason this confusion exists is that there are tools that can _disassemble_ bytecode back into something readable:

- `javap` — the JDK's built-in disassembler. It shows you the JVM instructions inside a `.class` file. The output looks structured and somewhat readable, but it is not the original Java source — it is a representation of low-level JVM opcodes (e.g., `invokevirtual`, `aload_0`, `bipush`).
- Decompilers like **JD-GUI** or IntelliJ's built-in decompiler go further and try to reconstruct something that resembles the original Java source — but that is a reverse-engineering step, not something you get from the compiler itself.

Java bytecode is also deliberately _higher-level_ than machine code (it targets the JVM, an abstract virtual machine, not a specific CPU), which is why decompilation works as well as it does. Machine code is far harder to reverse-engineer meaningfully.

**Summary:**

| Output            | Human-readable? | Notes                                                       |
| ----------------- | --------------- | ----------------------------------------------------------- |
| `.java` source    | Yes             | What you write                                              |
| `.class` bytecode | No (binary)     | But `javap` can disassemble it to readable JVM instructions |
| JVM machine code  | No              | Generated by JIT at runtime, never written to a file        |

### Does JavaScript have a human-readable layer after JIT?

No — JIT-compiled machine code in V8 is never exposed as a file or inspectable artifact. It lives in memory, runs, and is discarded. There is no `javap` equivalent for it.

But JavaScript has something different that serves a related developer need: **source maps**.

When Babel/SWC/esbuild transpile and bundle your code, the output JS is often minified and unrecognisable. Source maps (`.map` files) are companion files that record a mapping from every position in the output bundle back to the original source file and line. Browser DevTools use them to show you your original TypeScript/JSX in the debugger even though what's actually executing in the engine is the transpiled, minified bundle.

Source maps are not about JIT output — they bridge the gap between your authored source and the transpiled JS bundle, one level before the engine even runs it.

**Summary:**

| Layer                   | Human-readable?   | Developer tool                   |
| ----------------------- | ----------------- | -------------------------------- |
| Original `.ts`/`.tsx`   | Yes               | What you write                   |
| Transpiled `.js` bundle | Mostly (minified) | Source maps map back to original |
| V8 JIT machine code     | No                | Not exposed, no tool for it      |

### On calling Babel/SWC "compilers"

You're right that they're widely called compilers — and that's not wrong in the broad sense. In computer science, "compiler" just means "a program that transforms code from one representation to another". By that definition, Babel is a compiler.

"Transpiler" is the more precise term when the input and output are at the same level of abstraction (JavaScript → JavaScript, TypeScript → JavaScript). It's a subset of compilers.

In practice, both terms are used interchangeably in the JavaScript ecosystem. The important distinction to hold is: these tools transform source-to-source, they do not produce executable machine code.

## Transpilers: Babel, SWC, esbuild — A Different Kind of Compilation

If you come from a Next.js or Vite background, you've seen tools like **Babel**, **SWC**, and **esbuild**. These are easy to confuse with the Java-style compiler, but they do something fundamentally different.

### What they do: JS → JS (transpilation)

These tools are **transpilers** — they compile from one _high-level_ language to another _high-level_ language. The output is still JavaScript that a human can read. They do **not** produce machine code or bytecode — that job still belongs to the browser's JS engine.

Think of transpilation as translation between dialects, not translation to machine language.

### What each tool specifically does

| Tool        | Used in             | What it does                                                                                                                                                |
| ----------- | ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Babel**   | older setups, CRA   | Converts modern JS (ES2022+) → older JS (ES5) for browser compatibility. Also strips TypeScript types and transforms JSX into `React.createElement()` calls |
| **SWC**     | Next.js (default)   | Same job as Babel but written in Rust — much faster. Transpiles modern JS/TS → target JS version, transforms JSX                                            |
| **esbuild** | Vite, many bundlers | Extremely fast transpiler + bundler. Does transpilation AND resolves/combines `import` statements across files                                              |

### What they specifically handle

- **JS version targeting** — write `?.` optional chaining or `??` nullish coalescing, get output that works in older browsers
- **TypeScript stripping** — remove all type annotations (`:string`, `interface`, etc.) since browsers can't run TypeScript
- **JSX transformation** — convert `<Button />` syntax into plain JS function calls
- **Bundling** — resolve `import`/`require` statements and combine many files into fewer files (esbuild, Rollup, Webpack do this)
- **Tree shaking** — remove code that is imported but never used, making bundles smaller

### What they do NOT do

- They do **not** check types (that's TypeScript's `tsc`, a separate step)
- They do **not** produce machine code (the browser still does that at runtime)
- They do **not** enforce logic correctness — they only transform syntax

### The full pipeline in a Next.js project

```
.tsx source
  → [SWC] strips types, transforms JSX, converts to target JS
  → .js bundle (still human-readable JavaScript)
  → [browser's V8 JIT] compiles to machine code
  → CPU executes
```

### One-sentence summary

> Babel/SWC/esbuild translate between JavaScript dialects and handle syntax sugar — the browser's JS engine is still the one that does the real compilation to machine code.

## Core Difference in Typing

**Java (Static Typing)** — You must declare the data type of every variable (e.g., `int a`). The type cannot change later. This makes Java strongly typed.

**JavaScript (Dynamic Typing)** — Variables can hold any type at any time (e.g., `var a = 10; a = "hello"`). This makes JavaScript loosely typed.

## Data Types in Each Language

### Java Data Types

**Primitive Types** — fixed-size, strictly typed:

- `byte`, `short`, `int`, `long`
- `float`, `double`
- `char`
- `boolean`

**Reference Types:**

- Classes (e.g., `String`, custom classes)
- Arrays
- Interfaces
- Enums

Java enforces type safety at compile time, preventing accidental type mixing.

### JavaScript Data Types

**Primitive Types:**

- `number`
- `string`
- `boolean`
- `null`
- `undefined`
- `bigint`
- `symbol`

**Objects:**

- Arrays
- Functions (functions are objects in JS)
- Plain objects

JavaScript variables can switch between these types freely.

| -------------------- | --------------------------- | -------------------------------------- |
| Typing               | Static, strong              | Dynamic, weak/loose                    |
| Variable Declaration | Must specify type (`int x`) | `var`, `let`, `const` (no type needed) |
| Type Checking        | Compile-time                | Runtime                                |
| Primitive Types      | 8 fixed primitives          | 7 primitives (flexible)                |
| Objects              | Class-based OOP             | Prototype-based OOP                    |
| Function Type        | Methods inside classes      | Functions are first-class objects      |

---

## Primitives, Wrapper Types, Classes & Objects

### The Core Insight

Java and JavaScript both have "classes", "objects", and "wrappers", but they exist at different layers of the system.

**Java:**

- Classes are the **blueprint** level
- Objects are the **instance** level
- Wrapper classes exist to turn primitives into objects

**JavaScript:**

- Objects are the **fundamental** level
- Prototypes are the **inheritance** level
- "Classes" are just a nicer way to configure prototypes
- Wrapper objects exist to temporarily give primitives methods

The words are the same, but the architecture is different.

---

## Java: Primitives vs Wrapper Classes

Java has 8 primitive types: `int`, `double`, `boolean`, etc. These are:

- Not objects
- Stored on the stack
- Fast and lightweight

But Java also has **wrapper classes**:

| Primitive | Wrapper Class |
| --------- | ------------- |
| `int`     | `Integer`     |
| `double`  | `Double`      |
| `boolean` | `Boolean`     |
| `char`    | `Character`   |

**Why do they exist?**

Collections like `ArrayList` can only store objects, not primitives. Wrapper classes provide utility methods:

```java
int x = 5;
Integer y = Integer.valueOf(5);
y.toString(); // works
x.toString(); // does NOT work
```

> Java primitives are not classes, but they have class equivalents.
> When people say "Java classes wrap primitives", they mean: Java provides special classes whose job is to turn primitives into objects.

---

## JavaScript: Primitives vs Wrapper Objects

JavaScript primitives are also not objects, but here's the twist — JS **auto-wraps** primitives in temporary wrapper objects when you access properties:

```js
"hello".toUpperCase();
```

What actually happens:

1. JS creates a temporary `String` object
2. Calls `.toUpperCase()` on it
3. Discards the object

This is why primitives _feel_ like objects in JS, even though they aren't.

> When people say "JavaScript objects wrap primitives", they mean: JavaScript automatically creates temporary objects so primitives can use methods. This is a runtime behaviour, not something you explicitly code.

---

## Java Classes vs JavaScript "Classes"

### Java: Classes are real, fundamental, strict

A class is a **blueprint** for creating objects. It defines:

- fields (variables)
- methods
- constructors
- access modifiers
- inheritance rules

Objects are **instances** of classes. Everything except primitives is a class-based object.

```java
class Person {
    String name;
    void sayHi() { ... }
}
```

### JavaScript: Objects are fundamental, prototypes are the real mechanism

The fundamental building block is the **object**. Every object has an internal link to another object called its **prototype**. Inheritance happens through this prototype chain.

JavaScript added `class` syntax in ES6 (2015), but it is just **syntactic sugar** over prototypes:

```js
// What you write:
class Person {
  sayHi() { ... }
}

// What it actually is:
function Person() {}
Person.prototype.sayHi = function() {};
```

JavaScript "classes" are **not** real classes. They are a nicer way to work with prototypes.

### Java Classes vs JavaScript Classes — the real difference

| Feature         | Java Class                      | JavaScript "Class"               |
| --------------- | ------------------------------- | -------------------------------- |
| Real class?     | Yes                             | No (syntactic sugar)             |
| Type system     | Static, compile-time            | Dynamic, runtime                 |
| Inheritance     | Class-based                     | Prototype-based                  |
| Object creation | `new` creates instance of class | `new` calls constructor function |
| Methods         | Part of class definition        | Stored on prototype              |
| Safety          | Strong, enforced                | Loose, flexible                  |

---

## Are Objects the Same as Classes?

**Java:**

- Class → creates objects
- Objects ≠ classes

**JavaScript:**

- Function + prototype → creates objects
- Objects inherit from objects
- But objects are **NOT** classes
- And prototypes are **NOT** classes

JavaScript simply doesn't have the concept of a "class" in the Java sense.

Both Java classes and JavaScript "classes" sit at the **blueprint** level — they are both mechanisms for creating objects. But Java classes are a compile-time type system construct, while JavaScript "classes" are runtime prototype configuration wrapped in nicer syntax.

---

## The Wrapping Analogy

**Java:**

- Class = blueprint
- Object = house
- Primitive = raw material (brick)
- Wrapper class = a special blueprint that turns a brick into a house

**JavaScript:**

- Object = house
- Prototype = the architectural style the house inherits
- Primitive = raw material (brick)
- Wrapper object = a temporary scaffold built around the brick so you can work on it

**One-sentence summary:**

> Java wraps primitives using real **classes** because its type system is class-based.  
> JavaScript wraps primitives using temporary **objects** because its type system is object-based.

---

## Is Everything in Java a Class?

Not quite:

- Primitives are **NOT** classes
- Everything else is a class or object

## Is Everything in JavaScript an Object?

No:

- Primitives are **NOT** objects
- But they can behave like objects due to auto-boxing
- Functions **are** objects
- Arrays **are** objects
- Objects are the core building block
